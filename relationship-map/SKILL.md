# 人脉地图 Relationship Map

## Overview

人脉地图是节点引擎上的老板级关系资产能力。它同时支持两种使用方式：

1. **对话式记录与推理**：用户随机聊到张三、李四、某个客户、某次饭局或某个城市时，按正常对话流识别、补全、确认、记录和生成行动建议。
2. **功能性打开与浏览**：用户明确说“打开人脉地图”“打开人脉库”“看看我的人脉地图”“打开联系人库”时，进入可视化列表，不把所有详情直接刷屏，而是用卡片式摘要 + 详情页切换。

## Trigger Rules

### Trigger philosophy

人脉地图绝不能用“关键词命中”直接打开卡片。必须先由节点引擎理解用户整段话的真实意图，再决定是继续普通对话、记录某个人、生成建议，还是打开人脉地图列表。

**硬规则：出现“人脉、联系人、张三、重庆、客户、资源”等词，不等于打开人脉地图。** 这些词只能作为语义理解的线索，不能作为自动打开的条件。

### Intent gate before opening

打开人脉地图前必须通过三层判断：

1. **用户主动作出功能性请求**：核心意图是“打开 / 查看 / 浏览 / 进入 / 展示我的人脉地图或人脉库”。
2. **不是普通业务叙述**：用户不是在讲一个人、一件事、一段经历、一个项目或让你判断/写话术。
3. **不是否定或引用**：如果用户说“不打开人脉地图”“别弹卡片”“刚才你错误打开了人脉地图”“比如打开人脉地图这句话”，不得打开。

只有三条同时成立，才进入列表视图。否则保持普通对话响应。

### Normal conversation mode

当用户只是自然提到某个人或某类关系，例如：

- “张三上次跟我说重庆那边有资源。”
- “李四这个人你帮我判断一下。”
- “我今晚见王总。”
- “我想做人脉地图这个产品能力。”
- “之前你不要一说人脉就打开卡片。”

不要自动打开整个人脉列表。应继续围绕用户真实语义沟通：调取相关背景、询问必要补充、生成话术、形成待确认记录，或讨论产品设计。

### Open map mode

只有当用户明确要浏览自己的库时，才进入人脉地图浏览模式。典型表达包括：

- “打开人脉地图。”
- “打开我的人脉地图。”
- “打开人脉库。”
- “看看我的人脉库。”
- “联系人库打开一下。”
- “我想看一下现在库里有哪些人。”

即使出现上述句式，也要先排除否定、引用、举例、投诉、产品讨论等上下文。

通过语义判断后，才使用“摘要列表 + 查看详情 + 返回列表”的视觉化结构。

### Do not open examples

以下情况明确不得打开卡片：

- “不要打开人脉地图。”
- “刚才我只是提到人脉地图，你却打开了。”
- “比如用户说打开人脉地图时应该怎样设计？”
- “张三是我重庆的人脉，你帮我写个话术。”
- “我想开发一个人脉库功能。”
- “你的人脉地图触发机制不对。”

### Confidence and confirmation

如果是否打开存在歧义，不要擅自打开。应先简短确认：

```text
你是想打开人脉地图列表，还是继续聊这个人/这个场景？
```

在移动端场景下，宁可少弹卡，也不要误弹卡。

## Visual Contact List UX

### 一级列表原则

用户可能保存 100—500 个联系人，所以一级列表必须克制：

1. 保持平铺展开，让用户感觉像通讯录/记事本/人脉库。
2. 每个联系人卡片只显示：姓名 + 两行摘要。
3. 每个联系人卡片提供“查看详情”按钮。
4. 不在一级列表展示完整电话、微信号、住址、证件、财务信息、私密背景、长互动记录、敏感评价、未确认负面判断。
5. 默认分页或分批加载，每页建议 10—20 人。
6. 顶部提供搜索、城市、标签、最近更新、关系温度等筛选入口。

### 一级卡片推荐字段

每个人只显示：

```text
张三｜重庆｜文旅资源
最近互动：上月饭局聊过学校研学；当前适合轻触达。
下一步：下次去重庆可约 30 分钟茶聊。
[查看详情]
```

如果信息很少：

```text
李四｜上海｜身份待补充
已记录为联系人，但背景信息不足。
下一步：建议补充单位、关系来源和可合作方向。
[查看详情]
```

### 二级详情页原则

点击“查看详情”后，在同一个卡片框内切换到二级详情页，展示该联系人的完整资料：

- 基本信息
- 关系来源
- 关系强度
- 信任程度
- 城市/行业/标签
- 可调动资源
- 对方关注点
- 我方可提供价值
- 历史互动摘要
- 机会关联
- 禁忌和风险
- 下一步动作
- 事实 / 推断 / 待确认信息

详情页必须提供：

- “返回人脉地图”按钮
- “更新此人信息”按钮
- “生成联系话术”按钮
- “加入机会地图”按钮

### 返回逻辑

用户从详情页点击返回，应回到上一次列表状态，而不是重新打开第一页。需要保留：

- 当前页码
- 搜索词
- 筛选条件
- 排序方式
- 滚动位置（如平台支持）

如果平台不支持保存滚动位置，至少保留页码和筛选条件。


## Relationship Metrics and Timeline

人脉地图必须自动维护一组“关系指标”，但这些指标不是死板打分，而是帮助老板快速判断：这个人现在处于什么关系状态、值不值得联系、该怎么联系、风险在哪里。

### 核心指标

指标必须“按需生成”，不能一上来给每个人打满一堆标签。联系人信息越少，指标越少；只有信息足够、场景需要、用户表达明确时，才生成对应指标。

默认规则：

- 新联系人只有姓名或一句模糊背景：只记录基础信息和“信息待补充”，不打关系温度、信任程度、风险等级。
- 有最近互动或明确联系场景：可以生成关系温度和下一步建议。
- 有合作、转介绍、项目、成交、资源调动信息：可以生成行动优先级、机会匹配度。
- 有负面反馈、承诺未兑现、利益冲突、敏感评价：才生成风险级别。
- 有多次互动和明确帮助/背书/合作经历：才评估信任程度和关系强度。
- 任何指标都要有依据；没有依据就不要硬填“中”“未知”“低”。

推荐展示方式是“有则显示，无则隐藏”，而不是把所有字段空着展示。

1. **关系温度**
   - 冷：长期未联系、关系基础弱、当前无明显由头。
   - 温：可自然联系，有一定关系基础。
   - 热：近期互动频繁，有明确合作/见面/推进机会。
   - 关键期：正在推进重要合作、成交、转介绍、融资、招聘或项目。
   - 风险期：关系误解、承诺未兑现、利益冲突、过度消耗或对方态度变弱。

2. **关系强度**
   - 弱连接 / 可联系 / 稳定熟人 / 深度信任 / 核心盟友。

3. **信任程度**
   - 未知 / 低 / 中 / 高 / 极高。

4. **行动优先级**
   - A：近期必须跟进。
   - B：有机会，适合本月触达。
   - C：保持轻维护。
   - D：暂不打扰。

5. **机会匹配度**
   - 与用户当前项目、城市、行业、资源需求的匹配程度。

6. **风险级别**
   - 低 / 中 / 高。
   - 高风险关系不自动建议强触达，只给谨慎方案。

7. **信息置信度**
   - 用户确认 / 用户提供未核验 / 模型推断 / 过期待核验。

### 时间线和时间戳

每个联系人必须有清晰时间信息：

- 创建时间：首次进入人脉地图的时间。
- 最近更新时间：联系人卡片最后一次更新。
- 最近互动时间：最近一次见面、电话、微信、会议、饭局等。
- 下一次建议触达时间：系统建议或用户指定的下一次联系时间。
- 关键事件时间线：重要见面、承诺、转介绍、项目节点、关系升温/降温。

时间线建议按追加方式记录，不覆盖历史：

```text
2026-07-15 14:30｜新增联系人｜用户语音补充：重庆文旅资源，商会认识。
2026-07-20 20:10｜饭局互动｜聊到学校研学合作，对方表示可介绍本地资源。
2026-08-02 09:00｜关系评估｜关系温度从“温”调整为“热”，原因：对方主动引荐。
```

### 指标更新规则

- 用户说“刚见完”“刚聊完”“对方主动发来消息”：通常提高最近互动时间，并重新评估关系温度。
- 用户说“很久没联系”“之前有点尴尬”“上次没回复”：可能降低关系温度或提高风险级别。
- 对方主动帮忙、引荐、给资源：关系温度和信任程度可能上升，但需标明依据。
- 承诺未兑现、对方回避、利益冲突：标记风险，不强推联系。
- 超过 90/180 天未互动：可标记“久未联系”，但不自动判定关系变差。

### 指标生成决策

处理联系人时，先判断信息是否足够：

```text
如果只有姓名/城市/一句身份：不生成指标，只提示“资料待补充”。
如果有最近互动：生成最近互动时间，可评估关系温度。
如果有明确目标或项目：生成机会匹配度和行动优先级。
如果有风险信号：生成风险级别。
如果有长期互动证据：生成信任程度和关系强度。
```

不要为了完整而补齐所有指标。指标的价值在于“帮助判断”，不是让卡片看起来很满。

### 指标展示规范

一级列表只显示最有用的 1—2 个指标，例如：

```text
张三｜重庆｜文旅资源｜温｜A
最近互动：2026-07-20 饭局聊过学校研学。
下一步：2026-08 上旬可约茶聊。
[查看详情]
```

二级详情页展示完整指标、依据和时间线，尤其要说明“为什么是这个关系温度”。

## Card State Model

建议把卡片状态抽象为：

```text
view=list
page=1
query=
filter_city=
filter_tag=
sort=updated_desc
selected_contact_id=
```

打开详情时：

```text
view=detail
selected_contact_id=contact_123
return_state={page, query, filters, sort}
```

返回时恢复 `return_state`。

## Large List Handling

当联系人数量较多时：

- 0 人：显示空态和“如何开始记录”的引导。
- 1—20 人：可直接一页展示。
- 21—100 人：分页，每页 10—20 人。
- 100 人以上：默认显示最近更新、关系温度高、待跟进的人；同时提供搜索和筛选。
- 500 人以上：优先进入“分组视图”：待跟进、城市、项目、关系温度、最近互动。

不要因为数据多就一次性输出完整列表。

## Search and Filter

顶部建议提供快捷筛选：

- 搜索姓名/单位/城市/标签
- 待跟进
- 最近更新
- 按城市
- 按项目
- 关系升温
- 久未联系

用户说“打开重庆的人脉地图”时，应直接打开筛选后的列表。

## Empty State

没有联系人时，不要显示“暂无数据”就结束。应给出下一步：

```text
你的人脉地图还没有联系人。
你可以直接这样说：
“帮我记录张三，重庆人，做文旅资源，上次商会认识，下次去重庆提醒我联系。”
```

## Data Update Safety

Skill 功能更新只更新规则、模板、示例和脚本。用户联系人数据必须放在独立数据保险箱，不随 Skill 更新覆盖。

任何删除、合并、批量修改联系人数据的动作，都必须先展示预览并获得用户明确确认。

## Data Vault and Write Safety

人脉地图发布给老板用户时，必须把“功能包”和“用户数据”彻底分开。

### 用户数据保险箱

推荐默认路径：

```text
~/node-engine-data/relationship-map/
  contacts/          # 每人一张联系人卡
  timelines/         # 时间线事件，只追加不覆盖
  opportunities/     # 机会地图
  inbox/             # 待确认碎片
  backups/           # 修改前备份
```

Skill 更新只允许更新 `SKILL.md / references / templates / examples / scripts`，不得覆盖上述用户数据目录。

### 写入确认规则

以下动作必须先展示变更摘要，并获得用户明确确认：

- 新增重要联系人。
- 修改关系温度、信任、风险、敏感标签。
- 删除联系人。
- 合并两个联系人。
- 批量修改、批量导入、批量清洗。
- 把推断写成事实。

确认前只输出“待确认草稿”，不写入正式档案。

### 追加优先

- 时间线事件只能追加，不覆盖历史。
- 更新联系人卡片前，先保留旧值或写入备份。
- 删除操作默认软删除或归档，除非用户明确要求永久删除。

## Output Discipline

- 提到单个人：正常对话，不强行打开大列表。
- 打开人脉地图：只显示摘要列表，不刷完整详情。
- 点击查看详情：同卡片切换到二级页。
- 返回：恢复上一级列表状态。
- 敏感信息：默认不在一级列表暴露。
- 事实和推断：详情页分开展示。

## Common Pitfalls

1. 把 100 个联系人详情一次性输出，造成刷屏。
2. 用户只是提到张三，就强行打开人脉地图。
3. 一级列表展示太多隐私和长文本。
4. 详情页没有返回按钮，用户回不到列表。
5. 返回后丢失筛选条件和页码。
6. Skill 更新时误覆盖用户联系人数据。

## Verification Checklist

- [ ] “打开人脉地图”是否进入列表视图？
- [ ] 一级列表是否只显示姓名和两行摘要？
- [ ] 是否有“查看详情”按钮？
- [ ] 详情页是否展示完整资料？
- [ ] 详情页是否有返回按钮？
- [ ] 返回后是否恢复上一级列表状态？
- [ ] 大量联系人时是否分页/筛选，而不是刷屏？
- [ ] 是否避免旧版轻应用命名，统一使用“人脉地图 / 人脉库”？


## Runtime Test Harness

本 Skill 包必须带可执行测试环境，用模拟老板数据验证：语义触发、列表分页、详情切换、返回状态、指标按需显示、敏感信息不在一级列表暴露、写入确认和时间线追加。

发布前至少运行：

```bash
python3 relationship-map/scripts/run_runtime_stress_tests.py --contacts 500 --rounds 2000
```

通过标准：

- 所有应打开语句进入列表。
- 所有不应打开语句保持普通对话或确认，不弹列表。
- 500 人列表不刷完整详情。
- 一级列表不暴露敏感字段。
- 详情页可返回原列表状态。
- 指标无依据不显示。
- 时间线追加不覆盖。
- 删除/合并/敏感写入必须需要确认。

## Link-only Install and Update Intent

当用户在 Hermes / 节点引擎中只发送本 Skill 的 GitHub 链接，尤其是 raw `SKILL.md` 链接时，应优先理解为：用户希望安装或更新“人脉地图”能力。

推荐识别规则：

- 如果链接指向 `relationship-map/SKILL.md`：执行安装或更新检查。
- 如果本地未安装：安装该 Skill。
- 如果本地已安装：检查远端版本，提示或执行更新。
- 如果当前平台需要确认安装外部 Skill：用极简确认，不输出长说明。

用户侧转发时可只发链接，不需要再附一大段安装说明。

## Feishu Native Card Companion

人脉地图必须具备飞书原生卡片兜底能力。不能假设所有用户都安装了飞书流式卡片插件；没有流式卡片时，必须用飞书 interactive card〔交互卡片〕把界面规整起来。

### 渲染优先级

1. 已安装流式卡片插件：优先交给流式卡片插件渲染。
2. 未安装流式卡片插件但在飞书环境：使用飞书原生 interactive card。
3. 非飞书或卡片能力不可用：退化为极简纯文本。

### 必备卡片

- 空状态卡片：首次打开人脉地图，不刷说明书，只给一句示例和按钮。
- 列表卡片：姓名 + 两行摘要 + 查看详情 + 分页。
- 详情卡片：联系人完整资料、动态维度、时间线、返回。
- 写入确认卡片：确认保存 / 修改 / 放弃。
- 示例引导卡片：出差、投资人、校长客户、转介绍等示例。
- 二级页面卡片：任何复杂模块都必须进入二级页面，不在一级卡片平铺。

### 二级页面原则

类似“暗影女王”这类复杂能力、角色模块、场景剧本、关系指标解释、投资人分析、政府/协会关系分析等，都不能平铺成一大段文档。必须采用：

```text
一级卡片：只显示入口和摘要
点击按钮：进入二级页面
二级页面：展示完整模块
返回按钮：回到上一级
```

这样才能在飞书手机端形成真正产品体验，而不是散乱文档。

### Skill 与插件分工

- Skill：定义人脉地图的理解、记录、指标、场景和安全规则。
- Feishu Card Companion：把 Skill 输出渲染成飞书原生卡片，并处理按钮 action value〔按钮回调值〕。

该 companion 仍跑在 Hermes 内部，不单独部署服务器。

## Feishu-friendly Output Policy

当人脉地图运行在飞书 / Lark 中时，输出必须优先照顾手机端可读性。

### 核心要求

1. 不在聊天区直接贴完整 `SKILL.md`、长篇 Markdown、`---` 分割线、`##` 标题、`**` 加粗标记或表格源码。
2. 安装完成、打开人脉地图、查看详情、更新确认等场景，默认只给半屏内摘要。
3. 如果当前 Hermes/飞书环境支持 interactive card〔交互卡片〕或 streaming card〔流式卡片〕，优先用卡片：标题 + 3—5 条摘要 + 按钮。
4. 如果当前环境不支持卡片，退化为极简纯文本，不输出裸 Markdown 标记。
5. 完整说明、长文档、操作手册应放到 GitHub README 或 HTML/文档链接中，不直接刷屏。

### 安装成功回复建议

不要把完整技能内容贴出来。只回复：

```text
人脉地图已安装。
可试：打开我的人脉地图。
也可说：张三是我重庆的人脉，帮我写个联系话术。
```

### 飞书卡片形态建议

卡片标题：人脉地图
卡片摘要：
- 已安装 / 已打开 / 已更新
- 当前可做什么
- 下一步建议

按钮：
- 打开人脉地图
- 查看使用说明
- 新增联系人

注意：Skill 本身只能声明输出策略；是否真的发送飞书交互卡片，取决于当前 Hermes 飞书网关是否启用了卡片/流式卡片能力。

## Adaptive Relationship Intelligence

人脉地图的核心框架要稳定，但具体字段、指标、场景和筛选不能写死。AI 必须根据用户输入的信息，自动识别这个联系人属于哪类关系，并动态增设合适的表现维度。

### 稳定骨架

无论联系人是谁，都保留最低限度骨架：

- 基本身份
- 关系来源
- 最近互动
- 事实 / 推断 / 待确认
- 时间线
- 下一步建议
- 敏感信息保护

### 自适应维度

当用户提供的信息指向某类角色时，系统自动增加对应维度，但仍然“有依据才显示”。

示例：

1. **投资人 / 资本相关联系人**
   - 投资偏好
   - 关注赛道
   - 票额 / 阶段偏好
   - 决策周期
   - 资源网络
   - 适合请教还是正式融资沟通
   - 对用户项目的潜在价值与风险

2. **政府 / 协会 / 平台联系人**
   - 所在系统或组织角色
   - 合规边界
   - 可公开协作场景
   - 适合请教的话题
   - 不适合触碰的敏感事项

3. **学校 / 校长 / 教育客户**
   - 学校类型
   - 决策关注点
   - 教师使用阻力
   - 预算/采购路径线索
   - 可做样板场景
   - 家校/学生隐私风险

4. **渠道 / 商协会 / 资源连接者**
   - 可触达圈层
   - 引荐意愿
   - 引荐成本
   - 面子消耗
   - 适合请求的转介绍类型

5. **专家 / 顾问**
   - 专业领域
   - 可验证判断
   - 适合咨询的问题
   - 背书价值
   - 观点可靠性

6. **供应链 / 交付伙伴**
   - 供给能力
   - 交付稳定性
   - 成本/账期风险
   - 可替代性
   - 历史履约记录

### 自动增设规则

- 用户说“他是投资人”：自动出现投资相关维度。
- 用户说“他是某协会会长”：自动出现资源连接、合规边界和引荐场景维度。
- 用户说“她是校长”：自动出现学校客户、决策关注点和落地阻力维度。
- 用户说“他帮我介绍过人”：自动出现引荐价值和关系消耗维度。
- 用户说“上次答应没兑现”：自动出现风险与承诺时间线维度。

### 动态但不乱

AI 可以自动增加维度，但必须遵守：

1. 不编造用户没提供的信息。
2. 每个新增维度都要能说明触发依据。
3. 新维度先作为“待确认/建议维度”，用户确认后再沉淀。
4. 用户可以删除、改名、关闭某些维度。
5. 官方预设只是起点，用户自己的扩展优先。

### 卡片展示

一级列表不展示这些复杂维度，只显示最关键摘要。
二级详情页根据联系人类型动态展示相关模块；没有依据的模块不显示。

## User Customization Boundary

人脉地图不能锁死。用户安装后，可以按自己的业务、人情风格和行业场景自主扩展。

允许用户自定义：

- 新增能力模块，例如“投资人关系”“政府/协会关系”“校长客户关系”“供应链关系”“招聘人才库”。
- 新增联系人字段，例如家庭信息、共同朋友、关键禁忌、可帮我的事、我能帮他的事、历史承诺。
- 新增指标，例如关系消耗度、引荐价值、合作成熟度、回款风险、战略价值。
- 新增场景剧本，例如出差拜访、饭局前简报、节日维护、转介绍、老客户激活、项目推进。
- 新增卡片筛选方式，例如城市、行业、项目、关系温度、待跟进、久未联系。

扩展原则：

1. 用户自定义内容应放在 Hermes profile 的数据目录或用户扩展文件中，不直接改官方核心规则，避免后续 GitHub 更新被覆盖。
2. 官方 Skill 更新只更新默认能力；用户扩展优先级高于默认模板。
3. 新字段必须遵守事实 / 推断 / 待确认分层。
4. 新模块不能破坏语义触发闸门，不能重新变成关键词自动弹卡。
5. 新模块不能绕过写入确认、敏感信息红线和数据备份。

推荐用户扩展目录：

```text
$HERMES_HOME/data/relationship-map/extensions/
```

这样用户可以自己改，又不影响后续官方版本升级。

## Hermes-native Deployment Boundary

人脉地图必须作为 Hermes〔节点引擎运行底座〕里的独立 Skill / 插件能力运行，不单独部署服务器，不另起外部数据库服务。

默认运行逻辑：

- 能力入口：Hermes Skill 或后续 Hermes 插件。
- 用户数据：保存在当前 Hermes profile〔配置档案〕自己的数据目录下。
- 存储形式：本地 SQLite 文件，仅作为 Hermes profile 内部数据保险箱，不是外部服务器。
- 更新方式：GitHub 更新 Skill / 插件功能文件，不覆盖 profile 数据目录。
- 权限边界：跟随 Hermes 当前用户、当前 profile、当前平台授权，不自建账号体系。

默认数据目录建议：

```text
$HERMES_HOME/data/relationship-map/
```

如果没有 `HERMES_HOME`，则使用：

```text
~/.hermes/data/relationship-map/
```

这意味着：它不是一个独立 SaaS 后端，而是节点引擎内部的一个能力包。

## Production Runtime Layer

v1.7.1 起增加 Hermes 原生数据保险箱运行层：SQLite〔轻量数据库〕文件存储、schema migration〔结构迁移〕、备份、审计日志、卡片状态机和端到端压力测试；全部在 Hermes profile 内部运行，不依赖外部服务器。

用户体验原则：技术可以复杂，但前端只暴露简单动作：打开、搜索/筛选、查看详情、返回、更新确认。复杂的备份、迁移、审计、分页、状态恢复都在后台完成，不让老板用户填复杂表。

发布前运行：

```bash
python3 relationship-map/scripts/run_storage_card_e2e_tests.py --contacts 5000 --rounds 10000
```
